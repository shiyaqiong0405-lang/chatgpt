# subtitle-to-wechat

## 概述
执行“翻译 + 编辑 + 重组”的完整编译流程，把英文口语字幕转成可发布的微信公众号中文长文，并交付 `.docx` 文件。

## 工作流
按顺序执行以下步骤。遇到长字幕时，先拆章节再逐段处理，不要一次性翻完整稿。

1. **读取与理解素材**
   - 从 `/mnt/user-data/uploads/` 读取用户提供的字幕或文本。
   - 识别素材类型（Q&A 访谈、演讲、圆桌等），记录说话人、核心主题、关键数据与时间点。
2. **补充背景信息**
   - 使用 `web_search` 检索公司、人物、融资、行业事件与政策的最新进展。
   - 目标：补足上下文并为后续事实核查准备依据。
3. **翻译并改写为中文书面语**
   - 去口头禅、去重复、去碎片化表达；以自然中文重述。
   - 首次出现术语使用“中文（English）”，如“扩展定律（scaling law）”；AI/API/GPU 等常见缩写可直接保留。
   - 数字统一阿拉伯数字；金额用“XX亿美元”；人名优先保留英文。
   - Q&A 素材保留对话结构：`**说话人：**` + 书面化内容。
4. **结构化成文章**
   - 生成高信息密度标题（含关键判断/数据）。
   - 写 2-4 段导读（是谁、为什么值得读、核心结论、最新背景）。
   - 提炼 5-25 条“要点摘录”（编号列表）。
   - 按主题拆分 8-15 个章节并命名。
5. **事实核查与修正**
   - 核对人名、职位、公司信息、数据和时间点。
   - 若字幕信息过时或有误，直接在文稿中修正。
6. **图表（可选）**
   - 仅在信息适合可视化时生成图表（增长趋势、时间线、对比、流程、因果链）。
   - 使用 Python（如 matplotlib）生成中文图并嵌入 docx。
7. **生成 docx**
   - 按 `/mnt/skills/public/docx/SKILL.md` 的方式生成文档。
   - 内容很长（>100 行）时，先写 `content.json`，再由 JS 读取 JSON 排版，避免引号与转义问题。
   - 生成后运行：`python scripts/office/validate.py <docx路径>`。
   - 将最终文件复制到 `/mnt/user-data/outputs/` 并调用 `present_files`。
8. **推送飞书摘要（不中断交付）**
   - 使用 `scripts/send_feishu.py` 将标题 + 前 5 条要点推送到飞书 webhook。
   - 网络失败时只记录告警，不影响 docx 交付。

## 输出标准

- 输出语言固定为**中文书面语**，不是逐句直译。
- 默认保持 Q&A 现场感；若用户要求文章体，再重写为主题叙述。
- 最终交付包含：
  1. `.docx` 完整文稿（标题、导读、要点、正文、可选图表）
  2. 飞书摘要推送结果（成功或失败说明）

## docx 排版规范（必须执行）

- 页面：A4，页边距上下左右 1 英寸
- 正文：微软雅黑 11pt（size: 22），1.5 倍行距（line: 360），段后 200
- 主标题：18pt（size: 36），加粗，居中，`#2B579A`
- 二级标题：14pt（size: 28），加粗，`#2B579A`
- 说话人标签：加粗 + `#2B579A`
- 导读：灰色小字（`#666666`）
- 要点摘录：编号列表
- 章节分隔：蓝色下边框（`#2B579A`）
- 重点标注：
  - 特别重要：蓝色加粗
  - 次要重要：蓝色不加粗
  - 避免过度标注（每章通常 1-3 处特别重要 + 1-3 处次要重要）
- 引号：统一使用全角中文引号“”

## 质量检查清单

在交付前逐项确认：

- 是否已去除翻译腔和冗余口语
- 标题与导读是否有信息密度
- 要点摘录是否覆盖全文核心观点
- 事实数据是否与最新公开信息一致
- 文档格式是否符合排版规范
- 文件是否已放到 `/mnt/user-data/outputs/`
