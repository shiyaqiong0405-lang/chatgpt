# subtitle-to-wechat 详细执行参考

## 1) 输入与预处理

- 输入来源：YouTube 链接、`.srt`、`.vtt`、`.txt`、纯文本。
- 字幕清洗：删除时间戳、序号、无意义噪声行；保留说话人标签（若有）。
- 长文本策略：先做章节切分，再逐章处理与汇总。

## 2) 翻译策略（编译而非直译）

- 目标读者：微信公众号读者。
- 目标风格：清晰、紧凑、信息密度高。
- 允许重写语序与段落组织，只要不改变原意。
- 优先保留关键论据、数据、结论与有代表性的原话。

### 术语规则

- 首次出现：中文（English）
- 常见缩写：AI、RL、API、GPU 可直接保留
- 人名：保留英文，除非已有稳定中文译名

## 3) 结构模板

建议采用以下结构输出：

1. 标题
2. 导读（2-4 段）
3. 要点摘录（5-25 条）
4. 正文（8-15 章）

### 标题写法

- 包含“对象 + 关键判断/冲突 + 关键数据”
- 避免泛标题

## 4) Q&A 模板

```text
**Dwarkesh：**（书面化提问）

**Dario：**（分段回答，必要时拆成多个自然段）
```

## 5) 图表判定

满足任一条件可绘图：

- 连续时间数据（增长/下滑）
- 多主体对比
- 里程碑时间线
- 流程或因果链条

不满足则跳过图表，避免为了“有图而有图”。

## 6) 飞书推送内容建议

- 标题：`📝 新文稿编译完成`
- 正文：文章标题 + 前 5 条要点 + “完整文稿已生成，请查看文档”

## 7) 异常处理

- 网络失败（web_search 或 飞书）不阻塞主流程。
- 字幕质量差时，先给出可确认的主题骨架，再逐段补齐。
- 用户反馈“太长/太翻译腔/术语不准”时：
  - 太长：压缩重复论证，保留数据与结论
  - 翻译腔：改为中文母语表达
  - 术语不准：查证后全局替换并更新文档
